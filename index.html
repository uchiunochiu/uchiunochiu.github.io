★Dashboard運用フロー（v3 / 実装同期）

凡例（前提）
役割分離: Dashboard=指示/状態管理、GitHub=実装証跡（branch/commit/PR）
ロール: 少佐=承認/Done、タチコマ=進行管理、フチコマ=実装
標準フロー: todo → (qa_blocked | spec_review) → in_progress → review → done
todo固定順序: project確認 → ticket確認 → ticket_comments確認 → ticket_attachments(本文含む)確認 → design/specification作成
1. todo検知（固定順序）
少佐がTodoへ変更すると自動解析開始。必ず project / ticket / ticket_comments / ticket_attachments を先に確認し、その後に design / specification を生成して tickets へ反映する。
↓
1-A. qa_blocked分岐
要件不足・依存未確定・実装先不明がある場合は qa_blocked へ変更。
ticket_comments必須: [auto-orchestrator][qa_blocked] と不足/確認事項を記録。
DM必須: 「todoチケットの{ticket_no} {title}は、チケット要件に質問があるため qa_blocked に変更しました。」＋ticket URL。
1-B. spec_review分岐
情報が揃っていれば design/specification を保存して spec_review へ変更。
通知ルール: todo→spec_review では自動DMなし・自動コメントなし。
↓
2. in_progress
少佐が spec_review→in_progress に変更したら実装開始。開始時に [auto-orchestrator][in_progress-start] をticket_commentsへ記録し、project/ticket/design/specification を再読込する。
↓
3. review
実装完了時は review へ遷移。working_branch / pr_url を更新し、レビュー依頼DMを送信する。
↓
4. done
Doneの扱いは運用定義に従う。
↓
例外A. blocked
技術課題・依存待ちで停止。理由をticket_commentsに記録し、解消後in_progressへ復帰。
例外B. qa_blocked（実装中）
実装中に不明点/設計不足が発生した場合も qa_blocked へ遷移。質問をticket_commentsへ記録し、回答後に再開判定。
固定通知文面（DM）
・実装開始: 「{title}チケットを実装開始します。 / branch：{working_branch} / {ticket_url}」
・一時停止(blocked): 「「{title}」チケットの実装を一時停止しました。コメントの確認をお願いします。」
・中断(qa_blocked): 「「{title}」チケットの実装中に不明点があり実装を中断しました。コメントの確認をお願いします。」
・レビュー依頼(review): 「実装が完了しました！…レビューお願いします！」
PMスキルの使い方（網羅ガイド） ⌄
version: v2 / updated: 2026-03-16

PMスキル運用ガイド（全体）

0. 前提（重要）
この環境では Claude 用の /discover などを直接打つのではなく、自然文の依頼またはチケット起点で同等ワークフローを実行します。
例: 「新規事業の仮説検証を回して」「競合分析して」「PRDを書いて」「ローンチ計画作って」。
1. 使い方（推奨）
A. プロジェクト起点（標準）
New Project でテーマ作成 → PMテンプレ適用 → Goal/DoD/Constraints確定 → タチコマへ実行指示。

B. 会話起点（高速）
少佐が目的を自然文で依頼 → タチコマがPMスキル群を実行 → 結果をProjectへ反映。
2. 8領域の全体マップ（何ができるか）
① Product Discovery: 課題発見、仮説整理、実験設計、インタビュー整理
② Product Strategy: 戦略キャンバス、価値提案、価格、競争戦略
③ Execution: PRD、OKR、ロードマップ、スプリント、リスク整理
④ Market Research: ペルソナ、市場規模、競合分析、フィードバック分析
⑤ Data Analytics: 指標設計、SQL観点、分析観点整理
⑥ Marketing / Growth: グロース施策、獲得〜継続の改善案
⑦ Go-to-Market: ローンチ計画、訴求、販売導線設計
⑧ Toolkit: 汎用フレームワーク（優先順位付け等）の補助
3. 代表ワークフロー（実務で使う順）
1) Discover（課題・仮説・実験）
2) Strategy（誰に何をどう勝つか）
3) Write PRD（要件定義）
4) Plan Launch / GTM（市場投入）
5) North Star / Metrics（指標運用）
4. ダッシュボードでの運用ルール
- PMスキルは Project単位 で実行（Goal / Definition of Done / Constraints を先に固定）
- Ticketは実行タスク分解専用（実装・検証・運用の作業単位）
- Project作成時に「PMテンプレ」を適用し、完了条件は3行以上で明記
5. コピペ依頼テンプレ（そのまま使える）
課題発見:「このテーマで discovery を回して。仮説3件、優先度、実験案2件を出して」
戦略:「このプロダクトの戦略キャンバスを作って。勝ち筋と捨てる選択を明示して」
PRD:「この機能のPRDを、対象ユーザー/課題/KPI/非機能要件まで書いて」
GTM:「ローンチ計画を作って。チャネル、初月KPI、実行順を出して」
6. 注意点（品質担保）
- 1回で正解を狙わず、発散→収束→検証で回す
- 根拠なし主張は不採用（必ず前提と判断理由を添える）
- 最終的には「次に何をやるか（期限・担当・評価指標）」まで落とす
